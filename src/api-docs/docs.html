Encrypted-at-the-application-layer banking API.

Reference documentation for the BankServer HTTP surface. Every operation is grouped under one of the 18 numbered flow tags so the call order is self-evident. Each operation also carries:

  • description — markdown including a bulleted Prerequisites list of operationIds that must run before this one.
  • x-call-order — an integer that orders operations within the flow tag. Tools that respect extension fields can sort by this.
  • x-depends-on — machine-readable mirror of the prerequisites list.
  • x-decrypted-request / x-decrypted-response — for encrypted ops, the JSON shape carried inside the inner envelope's data field (i.e. the operation's actual payload). The wire-level requestBody / responses schemas keep the literal wire envelope ({ sessionId, payload } in / { payload } out); see Cryptographic envelope below for the inner envelope.

Cryptographic envelope

After the bootstrap handshake (flow 01-Bootstrap), every encrypted endpoint uses two nested envelopes:

1. Wire envelope — what travels over HTTP:

// Request
{ "sessionId": "<uuid>", "payload": "<base64 AES-256-GCM ciphertext>" }

// Response
{ "payload": "<base64 AES-256-GCM ciphertext>" }

payload is the AES-256-GCM ciphertext of the inner envelope below, with sessionId bound as Additional Authenticated Data (AAD). Schemas: EnvelopedRequest and EnvelopedResponse.

2. Inner envelope — what payload decrypts to:

{ "data": <operation payload>, "nonce": "<32 hex chars>", "timestamp": <ms> }

Canonical schema: DecryptedEnvelope. The fields:

  • data — the operation's actual JSON body. Its concrete shape is declared per-operation by the x-decrypted-request / x-decrypted-response extensions.
  • nonce — 16 random bytes hex-encoded. Per-session anti-replay; the server tracks recently seen nonces and rejects duplicates with 400 Replay detected.
  • timestamp — client-clock ms since epoch. Rejected with 400 Stale request if |now - timestamp| exceeds the freshness window (TIMESTAMP_WINDOW_MS = 60 000 ms / 1 minute).

Errors come back through the same two-envelope pipeline — the inner data carries { success: false, error: { ... } } (ApiErrorEnvelope).

Clients must also send x-session-id: <uuid> on every encrypted request so the server can locate the AES key without parsing the body.

Plaintext exception — /security/session and /security/handshake skip both envelopes (the AES key isn't established yet); their request and response bodies are plain JSON.

Where to find the decrypted shape per operation

The wire-level requestBody / responses schemas are deliberately the opaque outer envelope. The shape of the data field inside the decrypted inner envelope is documented in three places, in increasing fidelity:

  1. The per-operation requestBody.description and responses[<2xx>].description always name and link the schema, e.g. "Decrypted envelope's data field conforms to OtpRequestInput."
  2. The live Swagger UI viewer at /api-docs/swagger-ui renders a dedicated Decrypted body / Decrypted response schema explorer panel under the envelope (driven by x-decrypted-*). Each panel's subtitle reminds you the rendered shape is the data field, not the entire decrypted envelope.
  3. The Redoc viewers (/api-docs/redoc, /api-docs/docs.html) follow the description link straight to the schema under #/components/schemas/<Name> in the sidebar.

Step-up (sensitive actions)

Sensitive operations (transfers, opening accounts, freezing cards, revoking passkeys, ...) require a fresh WebAuthn assertion bound to the specific action and parameters. The flow is:

  1. postWebauthnAuthenticationVerify — returns { actionToken }.
  2. The caller replays x-action-token: <token> on the sensitive endpoint within the TTL. The token is one-shot.

Each step-up gated operation declares the action name in its description and adds the ActionToken security requirement.

Common error shape

All errors come back wrapped in the encrypted envelope (or plaintext for bootstrap routes) with body:

{
  "success": false,
  "error": { "message": "...", "requestId": "...", "details": { } }
}
